\capitulo{5}{Aspectos importantes del desarrollo del proyecto}

En este capítulo se explican las partes más importantes del desarrollo del proyecto. No solo se cuenta lo que se hizo, sino también cómo fue avanzando todo poco a poco. Además, se explican algunas decisiones que hicimos y algunos problemas que aparecieron durante el proceso.

\section{Ciclo de vida del desarrollo}

El proyecto se desarrolló de forma iterativa. Esto significa que no se hizo todo de golpe, sino poco a poco, por sprints, probando cada parte y mejorándola según iban surgiendo nuevas necesidades.

Primero se realizó un análisis inicial del problema. Se estudió qué tipo de información podía obtenerse desde Blender y qué acciones del usuario podían ser útiles para analizar el proceso de modelado.

Después se creó una primera versión centrada únicamente en capturar datos. Esta primera prueba sirvió para comprobar que Blender podía registrar automáticamente la información sobre las acciones realizadas por el usuario.

Las primeras pruebas ayudaron a comprobar aspectos importantes como:

\begin{itemize}
    \item la detección de eventos mediante \textit{handlers},
    \item el acceso a información de la escena en tiempo real,
    \item y el almacenamiento ordenado de los datos.
\end{itemize}

Cuando esta primera parte ya funcionaba, el sistema fue mejorando mediante distintas versiones. En cada una de ellas se añadieron nuevas métricas, se ajustó el rendimiento y se empezaron a incluir formas de visualizar la información.

Con el tiempo vimos que no bastaba con guardar datos. También hacía falta una forma sencilla de entenderlos. Por eso se desarrolló un segundo \textit{addon}, pensado especialmente para analizar y mostrar la información recogida.

El proyecto evolucionó de una aplicación simple para registrar datos hasta transformarse en un sistema más sofisticado dedicado al estudio del proceso de creación de modelos 3D.

Además, durante el desarrollo vimos que separar el sistema en distintos módulos hacía mucho más sencillo mantener el código y añadir nuevas funcionalidades sin afectar al resto del proyecto.

\section{Análisis del problema y definición de métricas}

Lo primero que tuvimos que establecer fue qué datos eran más importantes. Blender genera muchos cambios mientras se modela, pero no todos son igual de útiles para entender cómo trabaja el usuario.

Después de analizar varias opciones, dividimos los datos en tres grupos principales:

\begin{itemize}
    \item \textbf{Datos temporales}: tienen que ver con el tiempo de trabajo y el ritmo al modelar.
    
    \item \textbf{Datos espaciales}: se refieren a la ubicación y el movimiento de los elementos en la escena.
    
    \item \textbf{Datos estructurales}: se centran en la geometría del modelo, incluyendo vértices, caras u otros modificadores.
\end{itemize}

Gracias a estas métricas podemos guardar información como:

\begin{itemize}
    \item cuánto tiempo pasa el usuario modelando,
    \item cuántos vértices añade o elimina,
    \item cuánto se mueve dentro de la escena,
    \item o cada cuánto cambia de herramienta.
\end{itemize}

Dentro del sistema, las métricas se organizan con identificadores. Esto nos permite clasificar automáticamente cada dato que se registra.

Un ejemplo simplificado de esta estructura es el siguiente:

\begin{verbatim}
ANALYSIS = {
    "A1": {"label":"Total time","typology":"Temporal"},
    "A4": {"label":"Movement Speed","typology":"Spatial"},
    "A12":{"label":"Model evolution","typology":"Strategy"}
}
\end{verbatim}

Las métricas utilizadas por el sistema se organizan mediante identificadores internos y categorías asociadas. Esto permite clasificar automáticamente los distintos tipos de análisis, separando métricas temporales, espaciales y estructurales.

Gracias a esta estructura modular, el sistema puede ampliar fácilmente nuevas métricas sin modificar el funcionamiento general del addon.


Por ejemplo, algunas métricas sirven para medir el tiempo de trabajo, mientras que otras se centran en el movimiento o en cómo va cambiando la geometría del modelo.

\subsection{Primer \textit{addon}: sistema de captura de datos}

El primer \textit{addon} es el encargado de vigilar lo que ocurre dentro de Blender mientras el usuario trabaja.

Este componente detecta automáticamente eventos importantes y guarda información relacionada con el estado de la escena.

Por ejemplo, el sistema puede detectar:

\begin{itemize}
    \item creación de objetos,
    \item cambios de modo,
    \item modificaciones geométricas,
    \item transformaciones espaciales,
    \item o cambios en coordenadas UV.
\end{itemize}

Toda esta información se guarda después en archivos CSV, para poder analizarla más adelante.

\subsection{Segundo \textit{addon}: sistema de análisis y visualización}

El segundo \textit{addon} funciona como una capa de análisis sobre los datos que ya se han guardado.

Este componente no solo se encarga de mostrar información visualmente, sino también de procesar automáticamente los datos registrados para generar métricas y detectar patrones relacionados con el comportamiento del usuario.

Su objetivo es que la información recogida sea más fácil de entender. Para ello, genera representaciones visuales en 3D y también dentro del propio entorno de Blender.

Por ejemplo, el usuario puede ver gráficas basadas en los datos temporales, espaciales y estructurales recogidos por el sistema o detectar qué zonas de la escena han tenido más actividad durante el modelado.

Para conseguirlo, este componente utiliza distintos métodos de representación:

\begin{itemize}
    \item generación de geometría tridimensional,
    \item uso de librerías externas como \textit{matplotlib},
    \item y creación dinámica de objetos dentro de Blender.
\end{itemize}

El \textit{addon} de análisis permite transformar una gran cantidad de eventos en información más comprensible.

\subsection{Pipeline de análisis de datos}

Una vez que el sistema guarda la información en archivos CSV, el segundo \textit{addon} se encarga de procesar esos datos para convertirlos en métricas y representaciones visuales.

El proceso general de análisis sigue varias etapas consecutivas:

\begin{enumerate}
    \item lectura de los archivos CSV generados durante las sesiones de modelado,
    
    \item procesamiento y filtrado de los datos registrados,
    
    \item cálculo automático de métricas temporales, espaciales y estructurales,
    
    \item generación de representaciones visuales y gráficos.
\end{enumerate}

Este flujo permitió separar claramente las fases de captura, procesamiento y visualización, evitando que las tareas de análisis afectaran directamente al rendimiento de Blender durante el modelado.

\subsection{Generación de visualizaciones}

Una de las partes más importantes del segundo \textit{addon} es la forma en la que se muestran los datos recogidos durante el modelado. La idea no era solo guardar información, sino también conseguir que fuera fácil de entender visualmente.

Para ello usamos distintos tipos de representaciones dependiendo del tipo de información que se quisiera analizar.

Por ejemplo, el sistema puede generar:

\begin{itemize}
    \item gráficas temporales usando \textit{matplotlib},
    
    \item representaciones espaciales directamente dentro del entorno 3D de Blender,
    
    \item gráficos tipo radar para comparar distintas métricas,
    
    \item y geometría generada proceduralmente para representar información de forma visual.
\end{itemize}

Gracias a estas visualizaciones es más fácil detectar patrones, entender cómo evoluciona la escena y analizar mejor el comportamiento del usuario durante el modelado.

Además, al estar integradas directamente dentro de Blender, el usuario puede explorar la información sin necesidad de utilizar programas externos.


\section{Decisiones técnicas relevantes}

Durante el desarrollo tuvimos que tomar varias decisiones técnicas importantes para que el sistema funcionara correctamente y fuera cómodo de usar.

\subsection{Decisión de almacenamiento: CSV frente a base de datos}

Una de las decisiones más importantes fue elegir cómo guardar los datos.

Aunque las bases de datos se encuentran en muchos estudios, en nuestro proyecto decidimos manejar archivos CSV.

La elección de archivos CSV fue adecuada para nuestro caso, ya que encaja con nuestra forma de procesar la información y facilita el seguimiento y la verificación de los datos obtenidos.

Un ejemplo simplificado de registro generado es el siguiente:

\begin{verbatim}
USER_ID,TimeStamp,UserX,UserY,UserZ,VertexDelta,ModeChanged
a1b2c3d4,12.34,1.2,0.5,3.1,5,1
\end{verbatim}

\subsection{Captura de eventos mediante handlers}

Para capturar los eventos usamos los \textit{handlers} de Blender.

Los \textit{handlers} funcionan como avisos automáticos. Cuando pasa algo dentro de Blender, el sistema puede reaccionar y guardar información sobre ese cambio.

Algunas líneas del siguiente texto muestran cómo Blender facilita el registro de funciones automáticamente vinculadas a eventos interiores de la escena:

\begin{verbatim}
bpy.app.handlers.depsgraph_update_post.append(operator_tracker)
\end{verbatim}

De esta manera, Blender puede identificar cambios en la escena sin ninguna acción adicional por parte del usuario.

\subsection{Uso de librerías de visualización}

Para mostrar los datos de forma visual combinamos dos formas de representación:

\begin{itemize}
    \item visualización tridimensional dentro de Blender,
    \item y uso de librerías externas como \textit{matplotlib}.
\end{itemize}

Con este tipo de gráficos, podemos hacer uso de la versatilidad de las gráficas tradicionales proporcionadas por \textit{matplotlib}, mientras que Blender nos proporciona un espacio 3D para trabajar con dichas gráficas.

Por ejemplo, con \textit{matplotlib}

\begin{verbatim}
plt.plot(x, y)
plt.show()
\end{verbatim}

Además, el sistema también puede generar representaciones tridimensionales directamente dentro de Blender.

La combinación entre gráficos tradicionales y visualización espacial permitió representar distintos tipos de métricas dependiendo del análisis que se quisiera realizar.

\section{Estrategias de eficiencia del sistema}

Una parte importante del desarrollo fue intentar que el sistema no afectara demasiado al rendimiento de Blender.

Para conseguirlo, se aplicaron varias estrategias de optimización.

\subsection{Registro basado en cambios}

En vez de guardar información todo el rato, el sistema solo registra datos cuando detecta que ha pasado algo relevante en la escena.

\begin{verbatim}
if not state_changed and not any_operator_flag:
    return None
\end{verbatim}

Con esto se reduce mucho la cantidad de datos generados.

\subsection{Uso de firmas de estado}

Para evitar guardar información repetida, el sistema crea una especie de “firma” del estado actual de la escena.

\begin{verbatim}
signature = (verts, ngons, tris, mode, uv_hash)
\end{verbatim}

Solo se guarda información nueva cuando esta firma cambia.

\subsection{Control de frecuencia mediante temporizador}

El sistema también usa un temporizador para controlar cada cuánto se actualiza.

\begin{verbatim}
return 0.25
\end{verbatim}

Esto quiere decir que el sistema se actualiza unas cuatro veces por segundo, reduciendo así el impacto sobre el rendimiento general de Blender.

\section{Problemas encontrados y soluciones}

Durante el desarrollo aparecieron varios problemas técnicos. Algunos eran esperables, pero otros surgieron al probar el sistema dentro de Blender.

\subsection{Exceso de datos generados}

\textbf{Problema:} al principio el sistema guardaba demasiada información.

\textbf{Solución:} se añadió un sistema de registro basado en cambios y se eliminaron registros duplicados.

\subsection{Detección de eventos complejos}

\textbf{Problema:} algunos eventos de Blender no se podían detectar directamente.

\textbf{Solución:} se combinaron dos estrategias: analizar los operadores usados por el usuario y revisar el estado interno de la escena.

\subsection{Impacto en el rendimiento}

\textbf{Problema:} ejecutar los \textit{handlers} con mucha frecuencia podía afectar al rendimiento de Blender.

\textbf{Solución:}

\begin{itemize}
    \item reducir la frecuencia de actualización,
    \item evitar cálculos innecesarios,
    \item y usar estructuras de almacenamiento ligeras.
\end{itemize}

\subsection{Representación de datos incompletos}

\textbf{Problema:} en algunas métricas aparecían valores incompletos o poco coherentes.

\textbf{Solución:} se añadieron comprobaciones y filtros antes de mostrar los datos en las visualizaciones.

\section{Validación del sistema}

Para comprobar que el sistema funcionaba correctamente se realizaron distintas pruebas dentro de Blender utilizando sesiones reales de modelado.

Durante estas pruebas se revisó que los eventos se detectaran correctamente, que los datos se guardaran de forma organizada y que las métricas y visualizaciones funcionaran como se esperaba dentro del entorno 3D.

También se realizaron pruebas ajustando la frecuencia con la que el sistema revisaba los cambios en la escena, para comprobar que Blender siguiera funcionando de forma fluida mientras se capturaban los datos.

Con estas pruebas, se pudo ajustar más bien el equilibrio entre la cantidad de datos recopilados y la facilidad del sistema al momento de realizar tareas cuando el usuario lo hacía.

Parte de la validación se llevó a cabo durante la actividad práctica efectuada en la EASD Burgos, que contó con el participación de 6 estudiantes del curso de Animación en 1º y 8 estudiantes del curso de Producto en 2º.

La ocasión permitió probar el sistema en entornos prácticos donde trabajar, comprobando así si el procedimiento de recopilación, almacenamiento y presentación de datos se ejecutaba correctamente durante varias sesiones de modelado.

\section{Conclusiones del desarrollo}

El desarrollo del sistema nos permitió comprobar que es posible capturar y analizar automáticamente lo que hace un usuario dentro de un entorno de modelado 3D usando \textit{addons} para Blender.

Las decisiones tomadas durante el proyecto ayudaron a que la herramienta funcionara correctamente. La separación por módulos, la captura de eventos y la forma de mostrar los datos hicieron que el sistema tuviera mejor estructura y fuera fácil de usar.

Además, el uso de una arquitectura modular permitió desarrollar un sistema más flexible y preparado para futuras ampliaciones relacionadas con análisis de datos y visualización dentro de entornos 3D.
