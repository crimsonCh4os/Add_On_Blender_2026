\capitulo{5}{Aspectos relevantes del desarrollo del proyecto}

Este capítulo describe los aspectos más relevantes del proceso de desarrollo del sistema propuesto. En él se presentan las decisiones técnicas adoptadas durante las distintas fases del proyecto, así como las soluciones implementadas para abordar los principales retos encontrados.

A diferencia de los capítulos anteriores, centrados en los fundamentos teóricos y las herramientas utilizadas, este apartado se enfoca en la experiencia práctica obtenida durante la construcción del sistema. Se describen tanto el ciclo de vida seguido durante el desarrollo como los aspectos más significativos relacionados con el análisis del problema, el diseño de la arquitectura y la implementación de los addons desarrollados.

El objetivo de este capítulo es documentar las decisiones de diseño tomadas y justificar aquellas que han resultado especialmente relevantes para el correcto funcionamiento del sistema.

\section{Ciclo de vida del desarrollo}

El desarrollo del proyecto se ha llevado a cabo siguiendo un enfoque iterativo e incremental, ampliamente utilizado en ingeniería del software por su capacidad para adaptarse a cambios y validar soluciones de forma progresiva \cite{pressman2005}.

En una primera etapa se realizó un análisis preliminar del problema, identificando los requisitos principales del sistema y explorando las capacidades que ofrece la API de Blender para la detección de eventos y la extracción de información de la escena.

Posteriormente se desarrolló una primera versión experimental del sistema de captura de datos. Esta implementación inicial permitió comprobar la viabilidad técnica de funcionalidades clave, como:
\begin{itemize}
    \item La detección de eventos mediante handlers.
    \item El acceso a información de la escena en tiempo real.
    \item El almacenamiento estructurado de los datos generados.
\end{itemize}

Una vez validada esta fase inicial, el sistema evolucionó mediante sucesivas iteraciones en las que se incorporaron nuevas métricas, mejoras de rendimiento y refinamientos en la arquitectura del código.

En una fase posterior se identificó la necesidad de facilitar la interpretación de los datos recopilados, lo que condujo al desarrollo de un segundo addon orientado al análisis y la visualización de la información generada.

De este modo, el proyecto evolucionó desde una solución centrada únicamente en la captura de datos hacia un sistema completo de analíticas aplicado al modelado tridimensional, integrando tanto la recogida como el análisis visual de la información.

\section{Análisis del problema y definición de métricas}

Uno de los principales retos del proyecto consiste en determinar qué información resulta relevante para analizar el proceso de modelado 3D.

A partir del análisis realizado se definieron tres categorías principales de datos:

\begin{itemize}
    \item \textbf{Datos temporales}: permiten analizar la evolución temporal del proceso de modelado, incluyendo duración de tareas, pausas o ritmo de trabajo.
    
    \item \textbf{Datos espaciales}: describen la interacción del usuario dentro del entorno tridimensional, incluyendo posición, movimiento o proximidad entre elementos.
    
    \item \textbf{Datos estructurales}: caracterizan la geometría y evolución del modelo mediante información como vértices, caras, polígonos o modificadores.
\end{itemize}

Estas categorías se implementan en el sistema mediante un conjunto de métricas estructuradas que permiten organizar la información capturada y facilitar su análisis posterior.

Un ejemplo simplificado de esta estructura es el siguiente:

\begin{verbatim}
ANALYSIS = {
    "A1": {"label":"Total time","typology":"Temporal"},
    "A4": {"label":"Movement Speed","typology":"Spatial"},
    "A12":{"label":"Model evolution","typology":"Strategy"}
}
\end{verbatim}

Esta clasificación permite estructurar el análisis de forma coherente y facilita la generación de representaciones visuales posteriores.

\section{Diseño de la arquitectura del sistema}

Con el objetivo de garantizar una buena organización del código y facilitar el mantenimiento del sistema, la solución desarrollada se ha diseñado siguiendo una arquitectura modular compuesta por dos addons complementarios.

Esta arquitectura permite separar claramente las responsabilidades del sistema, distinguiendo entre la fase de captura de datos y la fase de análisis y visualización. De este modo, se reduce el acoplamiento entre componentes y se facilita la evolución independiente de cada uno de ellos.

\subsection{Primer addon: sistema de captura de datos}

El primer addon se encarga de monitorizar la interacción del usuario dentro del entorno de modelado de Blender.

Para ello se utilizan mecanismos de gestión de eventos proporcionados por la API de Blender, que permiten ejecutar código cuando se producen modificaciones en la escena.

Además de detectar eventos relevantes, este componente recopila información asociada al estado interno de la escena, incluyendo propiedades geométricas, transformaciones espaciales y datos temporales.

Toda esta información se almacena posteriormente en un formato estructurado para permitir su análisis posterior.

\subsection{Segundo addon: sistema de análisis y visualización}

El segundo addon actúa como una capa de análisis visual sobre los datos generados por el primero.

Su objetivo principal es facilitar la interpretación de la información recopilada mediante representaciones visuales tanto en 2D como en 3D dentro del propio entorno de Blender.

Para ello, este componente integra distintos mecanismos de representación, incluyendo:
\begin{itemize}
    \item Generación de geometría tridimensional.
    \item Uso de librerías externas como \textit{matplotlib}.
    \item Creación dinámica de objetos dentro de la escena.
\end{itemize}

Esta separación entre captura y análisis refuerza la modularidad del sistema y facilita su reutilización y escalabilidad.

\section{Decisiones técnicas relevantes}

Durante el desarrollo del sistema se tomaron diversas decisiones técnicas que resultaron fundamentales para garantizar el correcto funcionamiento de la herramienta.

\subsection{Decisión de almacenamiento: CSV frente a base de datos}

Una de las decisiones más relevantes fue la elección del formato de almacenamiento de los datos generados.

Aunque el uso de bases de datos relacionales o NoSQL es habitual en sistemas de análisis de datos, en este proyecto se optó por el uso de archivos CSV.

\textbf{Justificación:}
\begin{itemize}
    \item Simplicidad de implementación.
    \item Integración directa con Blender y Python.
    \item Compatibilidad con herramientas externas de análisis.
    \item Facilidad de exportación y reutilización de datos.
\end{itemize}

\textbf{Limitaciones:}
\begin{itemize}
    \item Falta de estructura jerárquica.
    \item Menor eficiencia en grandes volúmenes de datos.
    \item Ausencia de mecanismos avanzados de consulta.
\end{itemize}

El uso de CSV resulta habitual en procesos de análisis exploratorio de datos debido a su simplicidad y facilidad de uso \cite{data_analysis}.

Un ejemplo simplificado de registro generado es el siguiente:

\begin{verbatim}
USER_ID,TimeStamp,UserX,UserY,UserZ,VertexDelta,ModeChanged
a1b2c3d4,12.34,1.2,0.5,3.1,5,1
\end{verbatim}

\subsection{Captura de eventos mediante handlers}

La captura de eventos se implementa mediante el sistema de \textit{handlers} de Blender, que permite ejecutar funciones cuando se producen cambios dentro de la escena.

Por ejemplo:

\begin{verbatim}
bpy.app.handlers.depsgraph_update_post.append(operator_tracker)
\end{verbatim}

\textbf{Ventajas:}
\begin{itemize}
    \item Permiten capturar eventos en tiempo real.
    \item No requieren modificar el flujo de trabajo del usuario.
    \item Facilitan el acceso al estado actualizado de la escena.
\end{itemize}

\textbf{Problema principal:}

Los handlers pueden ejecutarse con alta frecuencia, lo que puede generar un impacto considerable en el rendimiento del sistema.

\subsection{Uso de librerías de visualización}

Para la representación visual de los datos se combinaron dos enfoques principales:
\begin{itemize}
    \item Visualización tridimensional dentro de Blender.
    \item Uso de librerías externas como \textit{matplotlib}.
\end{itemize}

Esta combinación permite integrar análisis clásico de datos con representación espacial dentro del entorno de modelado.

Por ejemplo, mediante \textit{matplotlib} se pueden generar gráficas convencionales de forma sencilla:

\begin{verbatim}
plt.plot(x, y)
plt.show()
\end{verbatim}

Asimismo, el sistema permite generar representaciones tridimensionales mediante geometría creada dinámicamente dentro de Blender.

\section{Estrategias de eficiencia del sistema}

Uno de los aspectos más críticos del desarrollo ha sido minimizar el impacto del sistema sobre el rendimiento de Blender.

Para ello se implementaron diversas estrategias de optimización.

\subsection{Registro basado en cambios}

En lugar de registrar información continuamente, el sistema únicamente genera registros cuando detecta cambios relevantes en el estado de la escena.

\begin{verbatim}
if not state_changed and not any_operator_flag:
    return None
\end{verbatim}

Este enfoque reduce significativamente el número de operaciones de escritura y disminuye el volumen de datos generados.

\subsection{Uso de firmas de estado}

Se implementó un mecanismo basado en firmas de estado para detectar modificaciones relevantes en la escena.

\begin{verbatim}
signature = (verts, ngons, tris, mode, uv_hash)
\end{verbatim}

Solo se registra información cuando esta firma cambia, evitando almacenar datos redundantes.

\subsection{Control de frecuencia mediante temporizador}

El sistema utiliza un temporizador con intervalo controlado para limitar la frecuencia de ejecución.

\begin{verbatim}
return 0.25
\end{verbatim}

Esto limita la actualización del sistema a cuatro ejecuciones por segundo, reduciendo el impacto sobre el rendimiento general de Blender.

\subsection{Filtrado de eventos irrelevantes}

También se implementó un filtrado de eventos con el objetivo de evitar la generación de información innecesaria o poco relevante para el análisis posterior.

\section{Problemas encontrados y soluciones}

Durante el desarrollo del proyecto se identificaron diversos problemas técnicos que requirieron soluciones específicas.

\subsection{Exceso de datos generados}

\textbf{Problema:} el sistema generaba inicialmente un volumen excesivo de información.

\textbf{Solución:} implementación de logging basado en cambios y eliminación de registros duplicados.

\subsection{Detección de eventos complejos}

\textbf{Problema:} determinados eventos de Blender no podían detectarse de forma directa.

\textbf{Solución:} combinación de análisis de operadores y comprobación del estado interno de la escena.

\subsection{Impacto en el rendimiento}

\textbf{Problema:} la ejecución frecuente de handlers afectaba al rendimiento del entorno de modelado.

\textbf{Solución:}
\begin{itemize}
    \item Reducción de frecuencia de actualización.
    \item Minimización de cálculos innecesarios.
    \item Uso de estructuras ligeras de almacenamiento.
\end{itemize}

\subsection{Representación de datos incompletos}

\textbf{Problema:} existencia de valores nulos o inconsistentes en algunas métricas.

\textbf{Solución:} filtrado y validación previa de los datos antes de su visualización.

\subsection{Escalado de métricas}

\textbf{Problema:} las diferentes magnitudes dificultaban la representación conjunta de los datos.

\textbf{Solución:} normalización de valores antes de su representación gráfica.

\section{Actividad de recogida de datos en la EASD Burgos}

Con el objetivo de validar el sistema desarrollado en un entorno educativo real y obtener datos de uso fuera de un contexto puramente experimental, se llevó a cabo una actividad práctica en la Escuela de Arte y Superior de Diseño de Burgos (EASD Burgos).

La actividad tuvo lugar el día 30 de abril y estuvo dirigida al alumnado del Ciclo Formativo de Grado Superior en Animación y del Grado en Diseño de Producto.

Durante la sesión se presentó el funcionamiento general del sistema desarrollado, explicando cómo Blender puede ampliarse mediante addons personalizados y cómo las analíticas de aprendizaje pueden aplicarse al estudio del comportamiento del usuario en entornos tridimensionales.

\subsection{Objetivos de la actividad}

La actividad perseguía varios objetivos:
\begin{itemize}
    \item Presentar una aplicación práctica del desarrollo software en Blender.
    \item Mostrar el funcionamiento de addons personalizados.
    \item Explicar el papel de las analíticas de aprendizaje.
    \item Recoger datos reales de interacción.
    \item Validar el funcionamiento del sistema en un contexto educativo.
\end{itemize}

\subsection{Desarrollo de la sesión}

La sesión se estructuró en dos partes principales.

En primer lugar, se realizó una introducción al desarrollo de herramientas dentro de Blender utilizando Python y la API del entorno.

Posteriormente, se presentó el sistema desarrollado en este Trabajo de Fin de Grado, mostrando tanto el addon de captura de datos como el sistema de análisis y visualización.

Tras esta explicación, el alumnado participó en una actividad práctica en la que pudo interactuar directamente con el sistema dentro del entorno 3D.

La actividad permitió recopilar información generada por usuarios reales durante sesiones de modelado.

Además, la actividad permitió acercar al alumnado conceptos relacionados con el desarrollo software, la visualización de datos y las analíticas aplicadas a entornos tridimensionales.

\section{Conclusiones del desarrollo}

El desarrollo del sistema ha permitido demostrar la viabilidad de capturar y analizar la interacción del usuario en entornos de modelado 3D mediante addons desarrollados para Blender.

Las decisiones adoptadas durante el proyecto, especialmente aquellas relacionadas con la captura eficiente de eventos, el almacenamiento de datos y la modularidad de la arquitectura, han resultado fundamentales para garantizar el correcto funcionamiento del sistema.

Asimismo, el proyecto pone de manifiesto el potencial de las analíticas aplicadas a entornos tridimensionales y abre nuevas posibilidades para futuras investigaciones relacionadas con el análisis de procesos creativos, la optimización de flujos de trabajo y el estudio del comportamiento del usuario en herramientas de modelado 3D.
