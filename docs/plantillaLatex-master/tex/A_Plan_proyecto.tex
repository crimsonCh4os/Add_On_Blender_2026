\apendice{Plan de Proyecto Software}

\section{Introducción}

El presente plan de proyecto tiene como objetivo establecer las bases organizativas y metodológicas para el desarrollo del sistema propuesto en este Trabajo de Fin de Grado. El sistema está compuesto por dos addons complementarios para Blender:

\begin{itemize}
    \item Un addon de recopilación de datos, encargado de registrar la actividad del usuario durante el proceso de modelado 3D.
    \item Un addon de análisis y visualización, orientado al procesamiento de la información registrada y a la representación visual de métricas dentro del entorno de Blender.
\end{itemize}

Este documento define la planificación temporal, los recursos necesarios, los criterios de viabilidad y las estrategias de seguimiento utilizadas durante el desarrollo del proyecto.

La definición de requisitos se realizó previamente al inicio del desarrollo. A partir de ese momento, el proyecto evolucionó siguiendo un enfoque iterativo e incremental basado en ciclos de trabajo de dos semanas (sprints), permitiendo incorporar funcionalidades progresivamente y validar cada componente del sistema.

El desarrollo se apoyó en principios inspirados en metodologías ágiles, utilizando iteraciones cortas, refactorización progresiva y mejora continua del sistema.

El repositorio GitHub actuó como herramienta central de gestión del proyecto, permitiendo el control de versiones, la trazabilidad del desarrollo y el seguimiento de la evolución del sistema mediante commits, incidencias (\textit{issues}) y versiones etiquetadas.

\section{Planificación temporal}

La planificación del proyecto se estructuró en sprints quincenales alineados con la evolución real del desarrollo. El seguimiento de cada sprint se realizó mediante el repositorio GitHub, utilizando \textit{commits}, \textit{issues} y versiones etiquetadas (\textit{tags}).

\begin{itemize}

    \item \textbf{Fase previa (Noviembre)}: estudio de viabilidad del proyecto. Se realizaron pruebas iniciales para evaluar la posibilidad de desarrollar un sistema de recopilación y análisis de datos en Blender, incluyendo el análisis de su API y la viabilidad técnica de la captura de eventos.

    \item \textbf{Sprint 0 (Enero)}: definición de requisitos, análisis del problema y estudio de la API de Blender.

    \item \textbf{Sprint 1 (Febrero - semanas 1-2)}: desarrollo del primer addon funcional. Implementación inicial de la captura de eventos y exportación a CSV.  
    \textit{Resultado: versión inicial funcional del sistema de recopilación de datos (tag: v0.1).}

    \item \textbf{Sprint 2 (Febrero - semanas 3-4)}: mejora del sistema de captura de datos y refinamiento de la estructura de almacenamiento CSV.

    \item \textbf{Sprint 3 (Marzo - semanas 1-2)}: desarrollo inicial del segundo addon orientado al análisis de datos y visualización de métricas.  
    \textit{Resultado: incorporación del sistema de análisis y visualización (tag: v0.2).}

    \item \textbf{Sprint 4 (Marzo - semanas 3-4)}: implementación de métricas temporales, espaciales y estructurales, así como primeras visualizaciones gráficas.

    \item \textbf{Sprint 5 (Abril - semanas 1-2)}: refactorización del sistema y mejora significativa del addon de recopilación.  
    \textit{Commit destacado: ``Refactorización del sistema de logging y detección de eventos (v0.4)''.}

    En este sprint se incorporaron mejoras relevantes:
    \begin{itemize}
        \item Soporte completo para \textit{Edit Mode} mediante \textit{bmesh}.
        \item Mejora en la detección de operadores (Shift+D, Alt+D, Ctrl+V y Merge).
        \item Detección robusta de coordenadas UV.
        \item Optimización del sistema de logging mediante registro basado en cambios.
        \item Incorporación de controles Start/Stop.
        \item Inclusión de paneles de depuración.
        \item Mejora en la generación y organización de archivos CSV.
    \end{itemize}

    \textit{Resultado: versión avanzada y estable del sistema de captura (tag: v0.4).}

    \item \textbf{Sprint 6 (Abril - semanas 3-4)}: mejora del segundo addon, integración de visualizaciones 2D y 3D, incorporación de gráficos mediante \textit{matplotlib} y optimización general del sistema.

    \item \textbf{Sprint 7 (Mayo - semanas 1-2)}: pruebas funcionales, validación del sistema completo y corrección de errores.

    \item \textbf{Sprint 8 (Mayo - semanas 3-4)}: documentación final y preparación de la entrega.  
    \textit{Resultado: versión final del sistema completo (tag: v1.0).}

\end{itemize}

La información reflejada en esta planificación se corresponde con la evolución real del repositorio del proyecto en GitHub.

\subsection{Viabilidad económica}

Aunque el proyecto se desarrolla en un entorno académico, es posible estimar su coste aproximado considerando un escenario profesional.

\subsubsection{Costes de software}

El proyecto utiliza principalmente herramientas de software libre, por lo que no implica costes directos de licencia:

\begin{itemize}
    \item Blender (GPL): 0 €
    \item Python (PSF License): 0 €
    \item GitHub (plan gratuito): 0 €
    \item Librerías utilizadas por el addon de análisis: 0 €
\end{itemize}

No obstante, el uso de un sistema operativo propietario como Windows puede implicar un coste aproximado de licencia de 150 €.

\subsubsection{Costes de hardware}

Se utilizó un equipo personal con un valor estimado de 1200 €. Considerando una amortización en 4 años (48 meses) y una duración del proyecto de aproximadamente 4 meses:

\[
\text{Coste hardware} = \frac{1200}{48} \times 4 = 100 \,€
\]

\subsubsection{Coste de desarrollo}

Para estimar el coste de desarrollo, se considera un perfil de desarrollador junior con un salario aproximado de 1200 € mensuales durante 4 meses:

\[
\text{Coste desarrollo} = 1200 \times 4 = 4800 \,€
\]

\subsubsection{Coste total estimado}

\[
\text{Coste total} = 150 + 100 + 4800 = 5050 \,€
\]

Por tanto, aunque el coste real del proyecto es reducido al tratarse de un entorno académico, su equivalente en un contexto profesional sería de aproximadamente 5050 €.

\section{Estudio de viabilidad}

\subsection{Viabilidad técnica}

El proyecto es técnicamente viable gracias a las capacidades que ofrece Blender mediante su API en Python.

El primer addon utiliza handlers y mecanismos de captura de eventos para registrar información relacionada con la actividad del usuario y el estado de la escena.

El segundo addon procesa posteriormente dichos datos mediante técnicas de análisis y representación visual, incorporando:
\begin{itemize}
    \item Cálculo de métricas.
    \item Generación de gráficos 2D.
    \item Visualizaciones tridimensionales dentro de Blender.
    \item Integración de librerías externas como \textit{matplotlib}.
\end{itemize}

La separación entre ambos addons facilita el mantenimiento del sistema y permite desacoplar completamente la fase de captura de la fase de análisis.

\subsection{Viabilidad legal}

Dado que Blender se distribuye bajo licencia GPL, los addons desarrollados para esta plataforma deben ser compatibles con dicha licencia.

Asimismo, las librerías utilizadas en el addon de análisis son compatibles con un uso académico y redistribuible.

El sistema incorpora mecanismos de consentimiento del usuario para la recopilación de datos, cumpliendo con la normativa vigente en materia de protección de datos.

\subsection{Análisis de riesgos}

Como principal riesgo del proyecto se identifica la complejidad de la API de Blender, especialmente en la detección de eventos en distintos modos de trabajo como \textit{Object Mode} y \textit{Edit Mode}.

Otro riesgo importante corresponde a la integración entre ambos addons y al procesamiento correcto de grandes volúmenes de datos generados durante sesiones largas de modelado.

Estos riesgos se mitigaron mediante:
\begin{itemize}
    \item Desarrollo iterativo.
    \item Refactorización progresiva.
    \item Validación continua.
    \item Optimización del sistema de logging.
    \item Uso de estructuras modulares desacopladas.
\end{itemize}
